#### [强制 §7] `system_storage.rs` 磁盘已用空间计算错误（可用空间虚高）

**问题**：`get_disk_info()` 用 `df -k /` 的 `Used` 列（`parts[2]`）作为 `disk_used`。macOS APFS 使用容器化多卷架构，`/` 挂载的是只读系统卷（Macintosh HD），其 `Used` 列只反映该单卷的已用空间（约 15 GB 系统文件），而用户数据在独立的数据卷（`/System/Volumes/Data`）。直接读 `Used` 列导致 `disk_used` 严重偏低（15 GB vs 真实 ~464 GB）。

连锁影响：
1. 前端 `freeBytes = disk_total - usedBytes` 虚高（916 GB vs 真实 ~467 GB），与 macOS 系统设置显示的可用空间（~466 GB）严重不符
2. `macos_remainder = max(0, disk_used - known_bytes)` 被钳制为 0（因 `disk_used=15GB` < `du` 扫描的 `known_bytes`），所有未扫描的空间（系统文件、APFS 快照、Mail、Messages、Photos 图库等）全部"消失"成可用空间

**修复**：改用 `total - available` 推算容器级已用空间。APFS 容器中 `1K-blocks`（总量）和 `Available`（可用）都是容器级共享值（所有卷相同），所以 `used = total - available` 正确反映整个容器的已用空间。读取 `parts[3]`（Available）而非 `parts[2]`（Used），并将 `parts.len()` 检查从 `>= 3` 改为 `>= 4`。

**修复前后对比**（以 931 GB APFS 容器为例）：

| 指标 | 修复前 | 修复后 | macOS 系统设置 |
|------|--------|--------|---------------|
| disk_used | 15 GB（仅系统卷） | 464 GB（容器级） | 464 GB |
| freeBytes | 916 GB（虚高） | 467 GB（准确） | 466 GB |
| macos_remainder | 0（被钳制） | ~264 GB（吸收未扫描项） | — |

**涉及文件**：[system_storage.rs](file:///Users/apple/Documents/github/tauri-app/src-tauri/src/clean_space/system_storage.rs) — `get_disk_info()` 函数。该修复同时影响 `scan_overview` 和 `scan_overview_stream`（两者都调用 `get_disk_info()`），以及前端的 `macos_remainder` 计算（`disk_used` 通过 `ScanStartPayload` 传给前端）。

#### [建议 §7] macOS 分类只显示单 item，无法下钻查看占用细节

**问题**：修复 `disk_used` 计算后，macOS remainder 正确吸收了所有未扫描的系统空间（可能几百 GB），但二级页面（`CategoryDetail`）只有一个 "macOS System" 单 item，用户无法看到到底是什么占用了这一半磁盘。这与 macOS 系统设置的存储视图形成对比——系统设置虽然也不细分，但用户期望能下钻查看。

**根因**：macOS 分类设计为单 item remainder（`macos_size = disk_used - known_bytes`），没有扫描系统路径进行细分。所有未扫描空间（APFS 快照、`/System`、`/Library`、`/private/var/vm`、Homebrew、`/usr`、`/bin` 等）全部堆进一个 item，用户只能看到一个巨大的数字却不知道来源。

**修复**：新建 `build_macos_category(disk_used, known_total)` 函数，并行扫描 6 个主要系统路径，构造 7 个子项的 macOS 分类：

| 子项 id | 名称 | 扫描路径 | 说明 |
|---------|------|----------|------|
| `macos_system` | macOS System | `/System` | 只读系统文件 |
| `macos_library` | System Library | `/Library` | 系统级框架和库 |
| `macos_vm` | Virtual Memory | `/private/var/vm` | sleepimage 和 swap 文件（需 root，可能返回 0） |
| `macos_logs` | System Logs | `/private/var/log` | 系统日志 |
| `macos_homebrew` | Homebrew | `/opt/homebrew` 或 `/usr/local` | Homebrew 包（Apple Silicon 用前者，Intel 用后者） |
| `macos_other` | Other System Files | `/` | 兜底项：APFS 快照、`/private/var/folders`、`/usr`、`/bin`、`/sbin` 等未扫描路径 |

`macos_total = disk_used - known_total`（吸收所有未扫描空间），`macos_other = macos_total - Σ(已扫描子项)`（吸收已扫描路径未覆盖的部分）。这样二级页面会显示 7 行，用户可以看到每个系统路径的占用，剩余的兜底项也明确标注为"APFS snapshots, temp folders, Unix binaries, and other system data"。

**接入点（3 处）**：
1. `scan_overview()`：原单 item macOS 分类构造替换为 `build_macos_category(disk_used, known_bytes)` 调用
2. `scan_overview_stream()`：7 个 `s.spawn` 线程改为返回 `total_bytes`，scope 后求和为 `known_total`，调用 `build_macos_category(disk_used, known_total)` 并 emit macOS 分类（最后 emit，因为依赖其他分类的总和）
3. 前端 [mac-os-remainder.ts](file:///Users/apple/Documents/github/tauri-app/src/features/clean-space/lib/mac-os-remainder.ts)：`incoming.id === "macos"` 时直接使用后端值（含 7 个子项），不重新计算单 item remainder。前 7 个分类 emit 时仍走原逻辑计算占位 macOS，最后 macOS emit 时用后端完整子项覆盖

**流式事件顺序**：`scan-start` → 7×`scan-category`（applications/downloads/documents/system_data/app_data/other_users/developer）→ `scan-category`（macOS，最后）→ `scan-complete`。macOS 必须最后 emit，因为它的 `total_bytes` 依赖其他 7 个分类的总和。

**涉及文件**：
- [system_storage.rs](file:///Users/apple/Documents/github/tauri-app/src-tauri/src/clean_space/system_storage.rs) — 新增 `build_macos_category()` 函数（line 329-403）；`scan_overview()` 和 `scan_overview_stream()` 接入
- [mac-os-remainder.ts](file:///Users/apple/Documents/github/tauri-app/src/features/clean-space/lib/mac-os-remainder.ts) — 新增 `incoming.id === "macos"` 早返回分支

#### [强制 §5] macOS 系统文件可被用户勾选删除（多层防护缺失）

**问题**：macOS 分类细化后，7 个子项（`/System`、`/Library`、`/private/var/vm`、`/private/var/log`、Homebrew、Other System Files）的 `risk_level` 都是 `Safe`，但 `command` 都是空字符串（只读系统文件不可清理）。前端的 `CategoryDetail` 没有区分"可清理"和"仅展示"的 item：

1. checkbox 可勾选——用户看到系统文件旁有可点击的 checkbox，会误以为能删除
2. "全选安全项"按钮（`selectAllSafe`）会选中所有 `risk_level === "safe"` 的 item，包括 macOS 系统文件子项
3. 点击"清理选中项"后，`executeBatchCleanup` 会把这些空 command 的 item 传给后端

**根因**：`StorageItem` 类型没有"可清理"标志，`CategoryDetail` 也没有区分有 command 和无 command 的 item。虽然后端 `execute_category_cleanup` 有空 command 跳过逻辑（line 49-53 `if item.command.is_empty() { failed += 1; continue; }`）和 `is_safe_command` 白名单（line 56-63），但前端的 UX 仍然让用户误以为系统文件可以删除——这是安全意识问题，不只是功能 bug。

**修复（5 层防御）**：

| 层级 | 位置 | 防护 |
|------|------|------|
| 1. UX | `CategoryDetail` checkbox 渲染 | `isCleanable = !!item.command`；不可清理的 item checkbox 灰显（`opacity-40` + `bg-muted`）+ `cursor-not-allowed` + 点击不触发 `toggleItem` + tooltip 提示"此项为系统/只读文件，不可清理" |
| 2. 批量选择 | `selectAllSafe` | 加 `&& i.command` 条件，"全选安全项"跳过没有 command 的 item |
| 3. 执行前过滤 | `handleCleanup` | `category.items.filter((i) => selected.has(i.id) && i.command)`，防御性过滤掉无 command 的 item（即使 selected 中意外包含） |
| 4. 后端空命令 | `execute_category_cleanup` | `if item.command.is_empty() { failed += 1; continue; }`（已有，line 49-53） |
| 5. 后端白名单 | `is_safe_command` | 非 `rm -rf`/`rm -f`/`find`/`docker system prune` 前缀的 command 拒绝（已有，line 56-63） |

**设计原则**：`command` 字段为空 = "仅展示，不可清理"。这既适用于 macOS 系统文件，也适用于其他分类中"查看但不可清理"的 item（如 Other Users 的 Shared Folder）。利用已有字段语义，不需要新增类型字段。

**涉及文件**：
- [CategoryDetail.tsx](file:///Users/apple/Documents/github/tauri-app/src/features/clean-space/components/CategoryDetail.tsx) — `selectAllSafe`、`handleCleanup`、checkbox 渲染三处加 `isCleanable` 判断
- [zh.json](file:///Users/apple/Documents/github/tauri-app/src/i18n/locales/zh.json) / [en.json](file:///Users/apple/Documents/github/tauri-app/src/i18n/locales/en.json) — 新增 `cleanSpace.detail.notCleanable` key
