# `<模块名>` Bug 台账

> **这是模板** — 复制到 `docs/modules/<id>/bugs.md` 后删除本行说明。
> 目录约定见 `docs/coding-standards.md` §11.2（落地后相对路径为 `../../coding-standards.md`）；由 `/fix` 工作流维护。
> 无 open bug 的模块可省略此文件；一旦记录第一条 bug，即从此模板起建。

---

## 🔴 Open（待修复）

> 每条一行，格式与项目 `[§X] 文件:行号 — 描述 — 建议 — 强制/建议` 输出约定对齐。
> 字段：ID · 严重度 · 现象 · 复现 · 定位 · 状态。

| ID | 严重度 | 现象 | 复现步骤 | 定位（文件:行号 / 猜测） | 记录日期 |
|----|--------|------|----------|--------------------------|----------|
| B-001 | 🔴高 / 🟡中 / 🟢低 | 用户可见的错误表现 | 1. … 2. … | `src/features/<id>/xxx.tsx:NN` 或"待定位" | YYYY-MM-DD |

---

## ✅ Closed（已修复，保留追溯）

> 修复后从 Open 移到此处，追加修复 commit 短哈希。`/fix` 工作流在提交后自动回写。

| ID | 现象 | 根因 | 修复 commit | 关闭日期 |
|----|------|------|-------------|----------|
| B-000 | 示例：切换语言后 toast 仍显示旧语言 | 顶层冻结 `t()` 文案 | `abc1234` | YYYY-MM-DD |

---

## 维护规则（与 `/fix` 工作流一致）

1. 用户报告 bug 或 AI 自查发现 → 追加到 **Open** 表，分配 `B-NNN` 顺序号。
2. `/fix` 修复完成、`pnpm run lint:fe && pnpm run test:critical` 通过后：
   - 该行从 Open 移到 **Closed**，填入修复 commit 短哈希。
   - 若该 bug 对应某 roadmap Backlog 项，同步勾选 `docs/modules/<id>/roadmap.md`。
3. 严重度定义：🔴高=数据丢失/崩溃/核心功能不可用；🟡中=功能受损但有 workaround；🟢低=体验/文案/边缘。
