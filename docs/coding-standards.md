# 编码规范

> 本文档基于当前仓库的既有实现总结，并吸收 `bettertolive` 文档里真正适合本项目的部分。
> 目标不是照搬别的项目规则，而是把这个仓库已经在做、且值得继续强化的约定写清楚。

## 0. 文档定位

本文档中的规则分三类：

- **强制**：新增代码和修改代码时必须遵守。
- **现状**：仓库中已经普遍采用，后续应继续保持。
- **建议**：有明确收益，但不作为一次改动中的硬门槛。

## 1. 通用约定

- **强制**：保持 TypeScript `strict` 模式通过，禁止引入未使用的局部变量、参数和导出。
- **现状**：`src/` 下模块优先使用 `@/` 别名导入。
- **强制**：模块按职责拆分，避免一个文件同时承担视图、状态、业务编排和平台调用。
- **建议**：注释只写职责边界、约束原因和非直观实现，避免解释显而易见的代码。
- **建议**：优先沿用仓库既有命名，不额外发明新后缀、新层级和新目录风格。

## 2. 前端目录与分层

### 2.1 Feature 组织

- **现状**：以前端功能域组织代码：`features/<domain>/page.tsx`、`feature.tsx`、`store.ts`、`hooks/`、`services/`、`components/`。
- **现状**：通用 UI 基元放 `components/ui/`，壳层布局放 `components/layout/`，跨功能共享逻辑放 `shared/`。
- **现状**：纯工具函数放 `src/lib/`，平台能力封装放 `src/platform/`。
- **强制**：新功能优先落在已有 feature 结构内，不在 `src/` 根部随意扩展平铺文件。

### 2.2 组件放置规则

- **强制**：跨功能复用的基础组件放 `components/ui/`。
- **强制**：单一功能模块内部复用的组件，优先放该 feature 的 `components/` 或 `shared/`。
- **建议**：只被一个页面使用的组件先就地放置，不提前抽象。
- **建议**：出现第二个明确使用方后，再决定是否上提到模块共享层；跨模块复用再上提到全局共享层。

### 2.3 命名约定

- **现状**：状态文件使用 `store.ts`。
- **现状**：业务编排文件使用 `*.use-cases.ts`。
- **现状**：外部接口适配文件使用 `*.repository.ts`。
- **现状**：控制器 hook 使用 `useXxxController`。
- **强制**：新增同类文件继续沿用这些命名，避免同一职责出现多套文件命名风格。

## 3. 前端状态、异步与职责边界

### 3.1 状态归属

- **现状**：使用 Zustand 管理功能级本地状态。
- **强制**：`store.ts` 只放状态和简单动作，不把复杂业务编排塞进 store。
- **强制**：业务规则放在 `services/*use-cases.ts`。
- **强制**：Tauri 调用、存储访问、外部接口适配放在 `services/*repository.ts` 或 `lib/tauri/commands/*`。
- **强制**：组件层尽量消费 controller 或 use case，不直接堆叠平台调用和业务细节。

### 3.2 异步安全

- **现状**：仓库已经在用 `useGuardedAsync` 这类模式防止重复触发。
- **强制**：会重复点击、重复提交、重复启动的异步动作，必须做重入保护。
- **强制**：`useEffect` 中注册的事件、定时器、订阅必须有清理逻辑。
- **强制**：旧请求覆盖新请求、卸载后继续写状态、重复触发导致状态错乱，视为真实缺陷，不是风格问题。

### 3.3 平台边界

- **现状**：平台能力统一封装在 `src/platform/`。
- **强制**：平台判断走 `canUseDesktopFeatures()`、`canUseTauriCommands()` 这类 helper，不把平台分支散落在 JSX 和业务代码里。
- **强制**：浏览器降级、桌面能力缺失、IPC 不可用这些分支必须在边界层处理，而不是留给页面自行兜底。

## 4. 国际化与用户反馈

### 4.1 国际化

- **现状**：语言资源维护在 `src/i18n/locales/`，通过 `react-i18next` 使用。
- **强制**：新增用户可见文案必须进入 i18n 资源，不直接硬编码到组件、toast、弹窗和菜单里。
- **现状**：应用在首屏渲染前等待 i18n 初始化，避免先显示 key 再闪成文案。
- **强制**：新增语言 key 时同步维护 `zh` 和 `en`，不要只补一侧。

### 4.2 反馈与容错

- **强制**：写操作要给结果反馈，成功和失败都要对用户可见。
- **强制**：如果用户输入被过滤、忽略、去重或拒绝，必须给出明确提示，禁止静默吞掉。
- **强制**：加载态、空状态、失败态至少要保证页面语义清晰，不能让用户面对无解释的空白区。

## 5. UI 与性能

- **现状**：优先复用 shadcn/ui 基础组件和 Lucide 图标。
- **现状**：类名拼接统一通过 `cn()` 处理。
- **强制**：基础样式优先沿用现有语义类和组件变体，不在调用处随意扩出一套新视觉语言。
- **现状**：大数据量视图已经使用虚拟化表格和虚拟化网格。
- **强制**：新增大列表、大表格、大量卡片时，优先评估虚拟化和渲染成本，不能默认全量渲染。
- **建议**：传给重组件的配置对象、回调和派生数据，优先使用 `useMemo` / `useCallback` 保持稳定引用。

## 6. Rust 后端约定

### 6.1 模块结构

- **现状**：Tauri 后端按领域拆分到 `src-tauri/src/<domain>/`。
- **现状**：常见分层包括 `commands.rs`、`types.rs`、`state.rs`、`mod.rs`，复杂模块再细分 `installer/`、`storage.rs`、`core.rs` 等。
- **强制**：新增后端能力优先进入现有领域模块，避免把不相关逻辑堆进 `main.rs` 或根级杂项文件。

### 6.2 阻塞与并发

- **现状**：文件系统、进程扫描、平台命令等阻塞操作通过 `spawn_blocking` 执行。
- **强制**：阻塞任务不得直接卡在 async 命令主路径中。
- **现状**：安装、卸载、升级、批处理等可变操作已经使用状态锁和取消标记保护。
- **强制**：新增可变后端操作必须明确并发策略：是否可重入、是否需要锁、是否支持取消、取消后状态如何恢复。
- **强制**：批量任务和取消接口按幂等思路设计，重复调用不能导致状态损坏。

### 6.3 错误边界

- **现状**：IPC 边界大量使用 `Result<T, String>`。
- **强制**：错误要在命令边界归一化，避免把底层细节直接泄漏成不可消费的异常形式。
- **建议**：错误信息至少要让前端能区分“权限问题 / 平台不支持 / 输入非法 / 通用失败”这类用户可见语义。

## 7. IPC 契约约定

这是当前文档从 `bettertolive` 吸收后，最值得强化的一部分。

### 7.1 单一事实来源

- **现状**：前端 IPC 契约集中维护在 `src/lib/tauri/contracts.ts`。
- **现状**：Rust 命令注册集中维护在 `src-tauri/src/commands.rs`。
- **现状**：前端类型封装位于 `src/lib/tauri/types/` 与 `src/lib/tauri/commands/`。
- **强制**：命令名、参数名、返回 DTO 字段必须在 TS 与 Rust 两侧保持一致。

### 7.2 改动清单

只要修改了一个 Tauri command，至少检查以下内容：

1. Rust 实现是否同步更新。
2. `src-tauri/src/commands.rs` 是否同步注册。
3. `src/lib/tauri/contracts.ts` 是否同步更新命令契约。
4. `src/lib/tauri/types/` 和 `src/lib/tauri/commands/` 是否同步更新。
5. 契约测试是否仍然通过。

- **强制**：跨 IPC 的改动不能只改一侧。
- **强制**：新增命令如果绕开现有契约体系，视为结构性回退。

## 8. 测试与质量门禁

- **现状**：前端测试使用 Vitest + jsdom + Testing Library。
- **现状**：后端使用 `cargo test`，并有 `cargo check`、`cargo clippy -- -D warnings`。
- **现状**：仓库提供统一脚本：`test:fe`、`test:be`、`build:fe`、`check:be`、`verify`。
- **现状**：pre-commit 会根据暂存文件范围执行前端、后端或脚本检查。
- **强制**：涉及 IPC、共享类型、复杂业务编排的改动，至少保证相关测试或检查链路通过。
- **建议**：优先补行为测试和契约测试，不优先补只验证实现细节的脆弱测试。

## 9. 提交规范

- **现状**：提交信息使用 Conventional Commits。
- **现状**：commit-msg hook 会校验类型、scope 和消息格式。
- **强制**：提交头必须可解析，禁止随意写自然语言标题。
- **现状**：commit body 的单行长度不得超过 500 个字符。

## 10. 建议立即补强的增量规范

这一节不是“仓库当前已经百分百达成”，而是参考 `bettertolive` 后，适合本项目立即收口执行的部分。

### 10.1 明确规则级别

- **建议**：后续所有规范文档都标注“强制 / 现状 / 建议”，避免把偏好混成硬门槛。

### 10.2 新增模块时的最小模板

- **建议**：前端新 feature 默认包含 `page.tsx`、`feature.tsx`、`store.ts`、`hooks/`、`services/`。
- **建议**：后端新 domain 默认至少明确 `mod.rs`、命令入口、状态或类型归属，而不是把逻辑直接散写到已有文件里。

### 10.3 评审聚焦真实风险

- **建议**：代码评审优先看这些问题，而不是样式噪音：
  - i18n 漏洞和硬编码文案。
  - Zustand 状态与异步副作用错位。
  - IPC 契约漂移。
  - 重复提交、并发执行、锁释放不完整。
  - 大列表渲染退化。
  - 平台能力判断散落导致的浏览器/桌面分支失真。

### 10.4 变更自检顺序

- **建议**：优先按“类型检查 -> 单测/契约测试 -> 前端构建 -> 后端检查”的顺序自检，先让工具暴露机械问题，再做人审。

## 11. 本次参考后明确不采纳的内容

以下内容在 `bettertolive` 中成立，但不适合直接搬到本仓库：

- **不采纳**：围绕 TanStack Query 的强绑定规范。本仓库当前并未以 Query 作为统一服务端状态层。
- **不采纳**：Specta 自动生成 bindings 的约束。本仓库当前使用手写契约和契约测试，不是同一链路。
- **不采纳**：过细的 i18n key 分类树。本仓库当前规模下，先保证“全部进入 i18n、双语同步、命名稳定”比堆叠层级更重要。
- **不采纳**：把所有偏好都定义成硬规则。规则过密会让文档失去可执行性。
