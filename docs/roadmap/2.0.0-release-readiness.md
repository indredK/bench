# Bench 2.0.0 功能完善与发布就绪规范

> 目标版本：2.0.0  
> 文档状态：发布门禁，全部完成前不得创建正式 `v2.0.0` tag  
> 当前基线：1.23.0  
> 配套 UX 规范：[2.0.0 用户体验完善与验收](./2.0.0-ux-readiness.md)

本文不是新功能愿望清单，而是把现有 Bench 能力补到可公开发布 2.0.0 的执行规范。模块细节仍以各自 `docs/modules/<id>/` 为真理源；发生冲突时按 `.cursorrules > AGENTS.md > ARCHITECTURE.md/coding-standards.md > 本文 > 模块 roadmap` 裁决。

## 1. AI 使用方法

后续 AI 必须按以下顺序执行，不得读完后一次性重写整个项目：

1. 先读 `AGENTS.md`、`.cursorrules`、`docs/ARCHITECTURE.md` §2、`docs/coding-standards.md`、`docs/UX-STANDARDS.md`、`docs/DECISIONS.md`。
2. 从本文选择一个工作包，继续读取对应模块的 `README.md`、`design.md`、`roadmap.md` 或专题审计。
3. 先用代码或失败测试证明缺口，再做最小完整修复；不得为了勾选 checklist 添加空壳接口、伪进度或成功空数组。
4. 同一 PR/任务只完成一个紧密相关工作包；IPC 变更同时更新 Rust、TS contract、wrapper、DTO 和测试。
5. 自动化通过不等于平台行为通过。要求真机证据的项目必须在对应 OS 完成，不得用 macOS 编译结果推断 Windows，反之亦然。
6. 只有“实现 + 自动化 + 目标平台证据 + 文档同步”四项齐全才能勾选。证据放在 PR/CI run/release issue，不把截图和日志堆进本文件。
7. 发现会导致数据损坏、凭据泄露、错误删除、错误更新、updater 签名缺失或产物集合不完整时停止发版。OS 包是否正式签名由 D-010 的显式模式决定，不能把 unsigned 伪装成 signed。

每次交付使用以下格式报告：

```markdown
工作包：Fxx
修改：文件与行为
自动化：命令 + 通过数量
平台证据：macOS / Windows 场景与结果
剩余风险：无 / 明确条目
文档：已更新的 roadmap/design/audit
```

## 2. 2.0.0 发布契约

### 2.1 默认平台范围

若发布负责人没有另行决定，按以下默认范围实施：

| 目标 | 2.0.0 状态 | 说明 |
|------|------------|------|
| macOS 14+ Apple Silicon | 目标平台 | ad-hoc 安装包 + 签名 updater；正式 notarization 延期 |
| macOS 14+ Intel | 目标平台 | 独立构建产物；功能语义与 Apple Silicon 一致 |
| Windows 11 x64 | 目标平台 | unsigned MSI/NSIS + 签名 updater；Authenticode 延期 |
| Linux | 不在 2.0.0 发布范围 | CI 可继续编译/测试；不发布 bundle，不承诺功能对等 |
| Browser preview | 仅开发模式 | 只验证可降级 UI，不作为正式产品形态 |

扩大最低系统范围前必须补对应 runner/真机证据；缩小范围必须在下载页、release notes 和应用内 capability 中一致说明。

### 2.2 功能平台矩阵

| 模块 | macOS | Windows | 2.0.0 契约 |
|------|:-----:|:-------:|--------------|
| Quick Launch | 必须 | 必须 | 同一 inventory、稳定 appId、真实启动、刷新/取消/partial 和 500+ 应用性能 |
| App Manager | 必须 | 必须 | inventory、图标、更新来源、批量操作和安装/更新安全边界 |
| Account Manager | 必须 | 必须 | 凭据、Session、probe、隔离 WebView、Deep Link 和失败语义对等；不支持的子能力显式 partial |
| Port Manager / Env Detector | 必须 | 必须 | 结构化平台 adapter、timeout、partial/error；不得解析本地化人类表格作为唯一来源 |
| Terminology / Token Calculator | 必须 | 必须 | 本地数据、搜索/计算、缓存 stale 和关键行为测试 |

`不适用` 不是失败，也不能显示为“关闭”或“空结果”。如果决定让 Clean Space 支持 Windows，必须先完成其 roadmap 中 `PlatformStorageScanner` 和清理白名单适配，再把矩阵改为“必须”。

### 2.3 明确不进入 2.0.0 的内容

- 云同步、维护者公共服务、AI Agent、TOTP、播放器、白噪音等新品类。
- Account Manager TLS/Canvas/WebGL 指纹模拟。
- Clean Space Undo、智能建议和定时提醒，除非已先完成对应安全设计。
- 使用频率/最近使用收集，除非先确定隐私、留存和清除策略。
- Linux 安装包与 Linux 功能对等。

这些内容不得挤占发布阻断项，也不得在 2.0.0 release notes 中预告为已经实现。

## 3. 当前基线与已知差距

当前值得保留：

- 应用清单已有单一 inventory、revision、single-flight、稳定 appId 和窄 IPC。
- Account Manager 已有 canonical Session map、Keyring/store 锁、一次性 auth ticket、严格 callback/state/origin、有界 probe retry 和 RefreshReport。
- 通用层已有 `RuntimeFeatureGate`、`FeatureLoadError`、`DestructiveConfirmDialog`、虚拟列表、结构化 Tauri 错误和 i18n 门禁。
- CI 已在 macOS/Windows/Linux 运行前后端测试和 Tauri debug build，并能构建 macOS/Windows updater 产物。
- release-please 已同步 `package.json`、Cargo workspace/lock 和 `tauri.conf.json` 版本。

当前代码/自动化阻断：

- Account Manager 的 per-origin local/session storage、IndexedDB、canonical Cookie expiry/partition key、capability DTO 和区域 retry 未完成；Windows 入口继续 gate。
- App Manager/Quick Launch 缺少两平台 fixture；500+ 应用性能验收未完成。
- updater 仍需用真实 Tauri updater 私钥跑通三目标 tag pipeline，并补损坏 manifest、错误签名、磁盘满和重启失败测试。
- 1.23.0 数据升级/回滚 fixture、全量视觉/a11y/键盘矩阵和 F09 版本切换未完成；仓库版本必须继续保持 1.23.0。

已延期但必须保留项目：macOS/Windows 真机安装与升级、WebView/Keyring/Deep Link 行为、Apple notarization 和 Windows Authenticode。延期项不得写成已验证，也不再作为本轮代码收口的伪阻断。

## 5. F00：平台与 capability 契约

**目标**：用户只能看到当前平台真实支持的功能；静态导航和动态子能力使用同一语义。

修改入口：

- `src/features/registry.tsx`、`src/features/types.ts`
- `src/platform/config.ts`、`src/platform/capabilities.ts`
- `src/components/common/RuntimeFeatureGate.tsx`
- 各 `src/features/*/feature.tsx` 与页面根组件
- `docs/roadmap/release-themes.md`

实现步骤：

1. 给 feature 元数据增加明确的发布范围；导航创建时按 `platforms` 过滤完全不适用模块，不能仅靠页面自行 gate。
2. 直达 URL 仍必须在路由根使用 `RuntimeFeatureGate`，避免隐藏导航后可绕过。
3. Account Manager 完成 Windows 后移除 `platforms: ["macos"]`；完成前保持 gate，不得直接删除限制后宣称支持。
4. 模块内部动态能力使用 `supported / partial / unsupported / failed`，并携带稳定 reason code。静态 feature 支持不能替代 WebView、Keyring、winget 等子能力探测。
5. macOS-only 模块在 Windows 不进入主导航；用户通过旧书签进入时展示本地化 unsupported 页面和可返回入口。
6. Browser preview 只 gate Tauri 能力，纯前端模块仍可使用；不得调用不存在的 IPC 后再用 toast 兜底。

建议契约：

```ts
type CapabilityState = "supported" | "partial" | "unsupported" | "failed"

type Capability = {
  state: CapabilityState
  reasonCode?: string
  detail?: string
}
```

验收：

- macOS/Windows/browser 三种 platform fixture 覆盖导航项目、直达路由和动态 action。
- unsupported 与 failed 不显示为空数据或关闭状态。
- `platforms` 元数据必须有注册表测试，防止新增 feature 忘记接 gate。

## 6. F01：数据迁移、启动恢复与降级

**目标**：从 1.x 升级到 2.0.0 不丢设置、账号、Session、分类和历史；损坏数据不会把应用卡死或静默重置。

修改入口：

- `src-tauri/src/lib.rs`、`src-tauri/src/bootstrap.rs`
- `src-tauri/src/account_manager/{state,storage,session}.rs`
- 各使用 `tauri-plugin-store`/localStorage 的 repository 和 store
- `src/components/common/StartupIssuesAlert.tsx`
- `docs/dev-prod-coexistence.md`

实现步骤：

1. 列出所有持久化键、文件和 schema version；每种数据明确 owner、迁移函数、最大尺寸和失败策略。
2. 迁移采用 `read old -> validate -> transform -> atomic write new -> publish memory`。迁移前创建同目录有上限备份；成功后按保留策略清理。
3. 迁移必须幂等；同一 1.x 数据重复启动不能生成重复账号、分类或记录。
4. 未知未来 schema fail-closed，不允许旧版本覆盖；提供导出备份或恢复说明。
5. 启动初始化失败进入 `StartupIssue`，页面显示受影响模块、可重试/打开数据目录/导出诊断；不能只 `eprintln!` 后继续显示空数据。
6. Dev/Prod 使用明确命名空间；Keyring 与 store 的环境隔离遵守 `dev-prod-coexistence.md`。
7. 所有秘密在日志、StartupIssue、诊断包中脱敏。

必须测试：

- 1.x 正常 fixture、缺字段、重复迁移、损坏 JSON/密文、只读目录、磁盘满、Keyring 拒绝、未来 schema。
- 迁移过程中强制失败后，旧文件仍可读取，新文件不出现半写状态。
- macOS Keychain 和 Windows Credential Manager 分别验证首次创建、并发、重启和卸载/重装策略。

## 7. F02：Account Manager 发布闭环

**目标**：完成 [Account Manager roadmap](../modules/account-manager/roadmap.md) 的 Phase 0-6；安全边界以 [design](../modules/account-manager/design.md) 和[生产审计](../modules/account-manager/audit-and-upgrade-2026-07-13.md)为准。

必须完成：

1. renderer 只能导出 sanitized 文件；后端拒绝不可跨设备恢复的 `encryptedFull`。未来如恢复完整导出，只能使用 passphrase + KDF + AEAD，文件不得包含本机 master key。
2. 实现按 origin 的 local/session storage 与 IndexedDB 捕获/恢复；Cookie schema 保留 canonical expiry 和 partition key。没有完整语义前 partitioned Cookie 继续 fail-closed。
3. App 根层建立唯一 Deep Link inbox：限制 URL 数量/大小、去重、顺序消费、冷启动恢复和敏感参数最短驻留。
4. Windows 接入第一个注册的 `tauri-plugin-single-instance` `deep-link` feature；自动化覆盖队列、多 URL 和重复 URL，冷/热启动及第二实例退出保留真机验证。
5. 网络代理密码更新使用 `keep / set / clear` 窄 DTO；Windows WebView proxy 有真实 provider 或明确 unsupported，任何失败不得回退直连。
6. 删除账号/站点返回逐资源 report：metadata、secret、session、binding、WebView/window/data dir 分别成功/失败；partial 对用户可见且可重试。
7. 密码仅按需 reveal，隐藏/切换账号/关闭弹窗/30 秒 TTL 时清内存；剪贴板继续条件清除。
8. 页面首载 skeleton、区域错误/retry、窄屏 Detail Sheet、500+ 账号虚拟化和批量刷新 partial UX 按配套 UX 文档实现。
9. 拆分 2000+ 行 `commands.rs`、过大的 controller/dialog 文件时按真实 owner 拆，不改 IPC 名称、不创建纯转发空壳。
10. 增加 `get_account_manager_capabilities` 或等价后端能力 DTO；Windows 入口只能在 Session/Keyring/WebView 基础链路可用后开放。

安全回归必须包含：ticket 重放、state 不匹配、callback host/port/path 替换、跨 origin 填充、并发 refresh、leader 取消、Cookie domain/path/partition、过期 Session、代理解密失败、导入炸弹和删除目录占用。

完成证据：专题审计 A-01 至 A-15 的 P0/P1 有测试并关闭；macOS/Windows 行为矩阵逐项通过，两个平台 capability 才能从 ⚠️ 改为 ✅。

## 9. F04：Clean Space 清理安全与平台收口

**目标**：macOS 清理结果可信、可解释、可部分失败；Windows 不出现虚假可用入口。

必须完成：

1. 在 feature 元数据和路由根声明 macOS-only；Windows 导航隐藏，直达显示 unsupported。
2. 为 `scan_storage_overview`、流式扫描、分类详情和 `execute_category_cleanup` 补 Rust 行为测试。
3. 消除 overview/stream 的路径分类构造重复，使同步快照和流式结果使用同一分类器。
4. 测试 Home 边界、`..`、symlink escape、系统根、Keychain/App State、只读文件、目录占用、跨卷、扫描中删除和磁盘数值变化。
5. 每个清理项返回 freedBytes、status 和 error；总释放量只能累加成功项。partial 后保留失败项并提供重试。
6. 外部 `du`/系统命令必须有可终止 timeout、输出上限和进程回收；阻塞遍历进入 `spawn_blocking`。
7. 清理记录有 schema/version/数量上限，不保存敏感完整路径；损坏记录不阻断扫描。

若实现 Windows provider，必须新增独立 `PlatformStorageScanner` 和 Windows 删除白名单，不能复用 macOS 路径字符串替换。

## 10. F05：开发工具、设置与本地模块收尾

只补现有能力，不开发 roadmap 中的远期新品类。

### Dev Toolbox / Port / Env

- 拆分 `dev-toolbox/page.tsx` 中 devtools/diagnostics/info 的真实职责，Tab 切换保留各自状态且不重复注册 listener。
- Port Manager 清理死代码，优化大 PID 树；Windows 使用结构化 API/固定 locale 输出，kill 重复调用幂等，PID 复用和权限失败有测试。
- Env Detector 对每个 provider 返回 supported/partial/failed；导出前脱敏 token、用户名和用户路径。补 timeout、取消、事件 taskId 和部分失败测试。
- 三者至少各有一个 controller/use-case 行为测试进入 `test:critical`。

### System Settings

- 所有启动读取从 `catch(console.error)` 改为 `value | unknown | error`；读取失败不能写成 `false`。
- 浏览器等枚举改为 canonical value，展示层 i18n；写操作严格 read/write/read。
- 权限拒绝、命令 timeout、系统版本不支持和回读不一致显示区域错误与系统设置入口。
- 保持 macOS-only，Windows 不进入导航；危险 quick action 继续二次确认。

### Terminology / Token Calculator / Hardware

- Terminology 大词表接入虚拟化并补搜索、复制、语言切换行为测试。
- Token Calculator 定价/汇率缓存显示更新时间与 stale；离线使用上次可信值并明确标识，不伪装实时。
- Hardware 保持 macOS-only；数据来源/更新时间可见。没有可追溯价格源时不展示“实时价格”。

## 11. F06：IPC、长任务、错误与资源约束

**目标**：所有现有模块在并发、取消和失败下具有一致的守恒 contract。

强制规则：

- IPC 返回统一 `{ code, message }` 或领域同构错误；新命令禁止 `Result<T, String>`，前端使用 `parseCommandError/getErrorMessage/translateError`。
- 长任务携带 `taskId`，共享 snapshot 同时携带 `revision`；前端忽略旧任务事件。
- 批量输入先去重并限制数量；输出满足 `total = succeeded + failed + cancelled`。
- 取消 token 可持续观察；进入不可取消区前更新状态。重复 cancel 幂等。
- 文件、响应体、归档、日志、导入数量、并发、队列和历史全部有上限。
- 外部进程 timeout 后回收进程树并释放 semaphore/文件锁；禁止把 renderer 输入拼进 shell。
- 启动/后台失败进入可见状态；生产日志不输出 password、cookie、token、OAuth query 或完整用户路径。

在 `src/lib/tauri/__tests__/contracts.test.ts` 增加所有新 DTO 字段检查；Rust 测试覆盖 poison/cancel/timeout/partial，前端覆盖旧事件不覆盖新任务。

## 12. F07：Updater、签名与供应链

**目标**：用户明确知道当前 OS 信任状态，Bench updater 可验证；缺少任一目标产物、updater 签名或校验文件时发布失败。

### Updater 功能

1. 实现自动检查策略：首次启动延迟、至少 24 小时间隔、网络错误指数退避、metered/offline 友好和用户开关；手动检查不受间隔限制。
2. 更新日志只渲染受控纯文本/安全 Markdown 子集，限制长度和链接协议；超长内容可滚动。
3. 下载保留 completed/total；未知总量显示 indeterminate。取消后可重试，安装阶段不可取消时明确提示。
4. 1.x -> 2.0.0 更新、同版本、损坏 `latest.json`、缺平台、错误签名、404、断网、磁盘满和重启失败都有测试。

### OS 与 updater 签名

- 当前默认 `BENCH_OS_SIGNING_MODE=unsigned`：macOS 使用 ad-hoc `"-"`，Windows 不做 Authenticode；Release 必须包含 `OS-SIGNING-NOTICE.txt` 和 `SHA256SUMS`，不得宣称 OS 已验证发布者。
- updater 始终要求 `TAURI_SIGNING_PRIVATE_KEY` 和 `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`；`.sig` 与三目标 `latest.json` 缺一即失败。
- 取得证书后切换为 `signed`：macOS 执行 Developer ID/notarize/staple，Windows 执行 Authenticode/时间戳。对应 Secrets 只在 signed 模式必需，流程见 D-010 和 workflow。
- updater 私钥与 OS 证书分开管理、最小权限、可轮换；任何 secret 不写入日志或 artifact。

### Release workflow fail-closed

- 移除正式产物收集中的 `|| true` 和静默可选分支。每个 target 必须存在 installer、updater bundle、`.sig` 和 manifest entry，否则 job 失败。
- `latest.json` 必须恰好包含 `darwin-aarch64`、`darwin-x86_64`、`windows-x86_64`，URL、signature 和文件一一对应。
- 生成并发布 SHA-256 checksums；推荐增加 GitHub artifact attestation/SBOM。`cargo audit`、`cargo deny` 和前端依赖审计在 release candidate 阶段必须通过或有明确、到期的例外。
- Release notes 不得为空，必须包含支持平台、最低版本、升级注意、已知限制和校验说明。

## 14. F09：版本切换、发布与回滚

1. 创建 code freeze，只接受 release blocker；确认所有模块 roadmap 的发布阻断已关闭或明确移出 2.0.0 范围。
2. 在 release-please 使用一次性、显式 `release-as: 2.0.0`（或维护者批准的等价机制）生成 Release PR；不要伪造 breaking commit 触发主版本。
3. 核对 Release PR 同步 `package.json`、`.release-please-manifest.json`、`src-tauri/Cargo.toml`、`Cargo.lock`、`tauri.conf.json` 和 `CHANGELOG.md`。
4. 需要真机验收时先发布 `v2.0.0-rc.1` 到非 latest 渠道，执行 F08 延期矩阵；发现数据/安全问题必须重新出 RC，不在原产物上覆盖。
5. 正式 tag 前验证 Git commit、tag、产物、latest.json 和 release notes 版本完全一致。
6. 发布后从 GitHub Release 执行一次干净安装和应用内更新，不使用 CI 工作目录产物代替。
7. 准备回滚：可撤下 `latest.json`/latest 标记阻止继续升级；已经升级的用户数据必须有备份和向前修复路径。不要让 1.x 打开并覆盖未知 2.0 schema。
8. 发布后 24 小时检查下载、签名、启动 issue 和 updater 错误；确认无 P0 后再关闭 release issue。
